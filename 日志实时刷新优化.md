日志刷新不实时是ELK默认配置的「缓冲+扫描间隔」导致的，核心优化2个点即可：**Filebeat缩短扫描间隔（实时发现新日志）+ Logstash关闭输出缓冲（实时转发）**，再补充ES/Kibana的实时检索配置，3步实现「秒级刷新」，完全适配你的7.14.0版本。

### 步骤1：修改Filebeat配置（缩短扫描间隔，实时采集）
打开 `./filebeat/filebeat.yml`，在 `filebeat.inputs` 下添加3个关键配置，覆盖默认的缓冲策略：
```yaml
strict.perms: false

filebeat.inputs:
  - type: log
    enabled: true
    paths:
      - /data/logs/*.log
    # 核心优化：实时采集配置
    scan_frequency: 1s  # 每秒扫描1次日志文件变化（默认10s，改1s实现近实时）
    close_inactive: 5m  # 5分钟无新日志才关闭文件句柄（避免频繁关闭导致漏采）
    harvester_buffer_size: 16384  # 增大缓冲，提升小日志实时性

output.logstash:
  hosts: ["logstash:5044"]
  pipelining: 0  # 关闭管道缓冲，日志即时发送（默认2，有延迟）

output.elasticsearch:
  enabled: false
```

### 步骤2：修改Logstash配置（关闭输出缓冲，实时转发）
打开 `./logstash/logstash.conf`，在 `output.elasticsearch` 下添加2个实时配置：
```conf
input {
  beats {
    port => 5044
    host => "0.0.0.0"
    ecs_compatibility => disabled
  }
}

output {
  elasticsearch {
    hosts => ["http://elasticsearch:9200"]
    user => "elastic"
    password => "lion123"
    index => "filebeat-search-%{+YYYY.MM.dd}"
    ssl => false
    # 核心优化：实时转发，关闭缓冲
    flush_size => 1  # 每条日志都立即刷新（默认500条才刷新，延迟根源）
    idle_flush_time => 1  # 1秒内无新日志也强制刷新（兜底）
  }
  stdout { codec => rubydebug }
}
```

### 步骤3：ES索引设置实时刷新（可选，Kibana秒级检索）
ES默认1秒自动刷新索引（`index.refresh_interval: 1s`），若仍觉得慢，可手动设置为500ms，在Kibana中执行一次即可（无需改配置文件）：
1. 打开Kibana → 「Dev Tools」（开发者工具）；
2. 输入以下命令，点击运行（绿色三角形）：
```http
PUT /filebeat-search-*/_settings
{
  "index.refresh_interval": "500ms"  # 索引每500ms刷新一次，Kibana可即时查到
}
```

### 步骤4：重启Filebeat+Logstash，验证实时性
```bash
# 进入ELK目录
cd /d/goCode/workspace/ELK
# 重启两个组件，加载新配置
docker-compose restart filebeat logstash
# 查看日志，确认无报错
docker logs -f filebeat
docker logs -f logstash
```

### 验证实时性：
在Windows `./logs` 下新建/追加日志内容（比如在`test.log`里新增一行），1-2秒内去Kibana「Discover」页面刷新，就能看到新日志，实现「秒级实时刷新」。

### 关键说明：
- 优化原理：默认配置为了性能，会缓冲多条日志再批量发送/刷新，改成「单条立即处理+短间隔扫描」，牺牲少量性能换实时性（测试/开发环境完全无压力）；
- 生产环境平衡：若日志量大（每秒千条以上），可把`flush_size`改10-50，`scan_frequency`改2-3s，兼顾实时性和性能。

按这个配置改完，日志刷新延迟会控制在1-2秒内，完全满足「实时查看」的需求～
